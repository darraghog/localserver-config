# TLS certificates

Certificates are generated by `scripts/setup-certs.sh` (separate from deployment).

## Why the browser says "unsafe"

1. **Untrusted CA** (most common) – The CA that signed the server cert is not in your system’s trust store. Add `certs/ca.pem` as a trusted root (see table below).
2. **Name mismatch** – The host in the URL is not in the cert. Use `https://127.0.0.1:8443`, `https://localhost:8443`, or the hostname/IP from setup-certs.

## Generate certs (once per host)

```bash
./scripts/setup-certs.sh
# Or with hostnames: ./scripts/setup-certs.sh darragh-pc 192.168.1.10
```

Creates `certs/ca.pem`, `certs/server.pem`, `certs/server-key.pem`.

## Trust CA on clients

Browsers use the system trust store. Add `certs/ca.pem` to the **machine you browse from** before visiting the HTTPS URLs. Copy it from the server if needed: `scp darragh-pc:~/localserver-config/certs/ca.pem .`

| OS | Command |
|----|---------|
| Linux | `sudo cp certs/ca.pem /usr/local/share/ca-certificates/localserver.crt && sudo update-ca-certificates` |
| macOS | `sudo security add-trusted-cert -d -r trustRoot -k /Library/Keychains/System.keychain certs/ca.pem` |
| Windows | See below |

**Windows (Chrome/Edge on same machine as WSL):** The Windows browser needs the CA trusted. From WSL, `certs/ca.pem` is at e.g. `~/localserver-config/certs/ca.pem`. In Windows Explorer, open `\\wsl$\Ubuntu\home\<user>\dev\localserver-config\certs\` (adjust distro/path) and copy `ca.pem` to your Desktop. Then: press Win+R → `certmgr.msc` → Trusted Root Certification Authorities → Certificates → right‑click → All Tasks → Import → select `ca.pem` → place in "Trusted Root Certification Authorities" → Finish. Restart the browser. (Firefox: Settings → Privacy & Security → Certificates → Import → select ca.pem.)

**Browser-specific:** Chrome and Edge use the OS trust store. Firefox uses its own.

## Troubleshooting

**HTTPS not working** – run on the server:
```bash
./scripts/check-tls.sh
```

| Check | Possible cause |
|-------|-----------------|
| Caddy not running | `cd compose/tls-proxy && podman compose up -d` |
| No certs | Run `./scripts/setup-certs.sh` first |
| Cert mismatch | Access using the IP or hostname in the cert (run setup-certs with that value) |
| Connection refused | Firewall blocking 8443/8444; open ports or `sudo ufw allow 8443,8444` |
| Cert invalid / untrusted | Trust `certs/ca.pem` on the client (see table above) |
| Windows browser + WSL same machine | Import `certs/ca.pem` into Windows Trusted Root CAs (see Windows section above) |
| ERR_SSL_PROTOCOL_ERROR | Use `https://` not `http://`; try `https://127.0.0.1:8443` (IPv4) instead of localhost; ensure Caddy is running; from Windows try WSL IP: run `hostname -I` in WSL and use that IP |
| darragh-pc doesn't resolve on Windows | Add to `C:\Windows\System32\drivers\etc\hosts` (as Admin): `192.168.86.237 darragh-pc` — or use `https://192.168.86.237:8443` directly |
| darragh-pc and IP both fail on Windows browser | **Simplest (same machine)**: Add `127.0.0.1 darragh-pc` to `C:\Windows\System32\drivers\etc\hosts` (edit as Admin). Then https://darragh-pc:8443 uses localhost, which works. **Port forward**: Run `scripts/setup-windows-port-forward.ps1` as Admin. **Different machine**: Add proxy bypass; ensure same network. |
